## 第 0 步：先建立心智模型（10 分钟必读）

### 0.1 AgentCore 是什么

Amazon Bedrock **AgentCore** 是一套「把 Agent 跑到生产、并持续把它变好」的托管服务。本实验只用到其中四块：

| 能力 | 一句话 | 解决的问题 | 主要 API |
|---|---|---|---|
| **Harness（托管运行）** | 只交「配置」，AgentCore 帮你跑 Agent 循环 | 不想自己写编排/部署容器 | `CreateHarness` / `InvokeHarness` |
| **Runtime（自带代码）** | 交「代码」，部署到无服务器运行时 | 要完全控制 Agent 逻辑 | `agentcore deploy` / `InvokeAgentRuntime` |
| **Evaluations（评估）** | 用「LLM 当裁判」给 Agent 行为打分 | 怎么客观知道 Agent 好不好 | `StartBatchEvaluation` |
| **Optimization（优化）** | 从评估结果反推一版更好的提示词 | 怎么自动地把 Agent 改好 | `StartRecommendation` |

### 0.2 这次要搭的闭环

```
   ┌─────────────────────────────────────────────────────────────┐
   │                                                             │
   ▼                                                             │
[创建/部署]  托管 Harness（模型 + 故意很弱的提示词 + 5 个工具）         │
   │         用 InvokeHarness「客户端工具循环」调用                   │
   │         （同一个 Agent 也部署一份 Runtime 镜像，专门用来评估）      │
   ▼                                                             │
[观测]      GenAI 轨迹 span + 消息内容 → CloudWatch Transaction Search │
   ▼                                                             │
[评估]      AgentCore Evaluations（批量、LLM 裁判）→ 基线分数           │
   ▼                                                             │
[优化]      Recommendations 分析轨迹 → 改进提示词 → 应用 + 重新部署       │
   ▼                                                             │
[验证]      重新评估 → 对比 → 「晋升」或（本次）「不晋升 + 回滚」 ──────────┘
```

> 💡 **一句话理解闭环的价值**：评估不是为了拿个分数好看，而是为了在「自动改进」之后，
> **有客观依据决定要不要上线这次改动**。本实验最精彩的地方，恰恰是优化**变差了**、而评估**把它拦住了**。

### 0.3 这个 Agent 是谁

一个 **Acme 网店客服**。它有 5 个工具（查订单 / 退货 / 查物流 / 打折 / 转人工），背后是一份
固定的订单数据（`ORD-1001` ~ `ORD-1005`）。我们故意给它一份**很弱的系统提示词**，留出可被优化的空间。

## 第 1 步：准备环境

### 1.1 你需要

- 一个 **AWS 账号**，并在 `us-west-2` **开通了 Anthropic Claude 模型访问**（Bedrock 控制台 → *Model access*）。
- 本机配置好 AWS 凭证（环境变量 / `~/.aws` / SSO 均可）。
- [`uv`](https://docs.astral.sh/uv/)（自动帮你装 Python 3.12 和依赖）。
- CloudWatch **Transaction Search** 已开启（评估服务靠它发现会话轨迹；`preflight.py` 会检查）。

### 1.2 安装依赖 + 体检

```bash
cd eval_opti
uv sync                       # 装依赖 + 拉 Python 3.12
uv run python preflight.py    # 体检：身份、Claude 访问、Transaction Search、boto3 能力 -> config.json
```

`preflight.py` 会写出 `config.json`。打开看一眼，确认这几项：

```jsonc
{
  "region": "us-west-2",
  "agent_model_id": "global.anthropic.claude-haiku-4-5-20251001-v1:0",  // 自动挑的最便宜的 Claude
  "transaction_search": "on (ACTIVE)",                                   // 必须是 ACTIVE
  "capabilities": { "harness_api": true, "evaluation_api": true, "optimization_api": true, ... }
}
```

> ⚠️ 如果 `preflight.py` 报 Bedrock 访问 DISABLED：去 Bedrock 控制台（us-west-2 → Model access）开通一个
> Anthropic Claude 模型，等状态变成「Access granted」后重跑。这是唯一会**硬阻塞**的前置条件。

> 💡 **想一想**：为什么评估服务需要 Transaction Search？因为评估是「读 Agent 跑过的真实轨迹来打分」，
> 而这些轨迹（span + 事件）先被写进 CloudWatch，Transaction Search 负责把它们索引出来供评估查询。

## 第 2 步：看懂这个 Agent——工具 + 故意写差的提示词

> 🎯 本步目标：在动云之前，先在本地彻底搞懂「Agent = 模型 + 提示词 + 工具」。

### 2.1 工具逻辑（确定性、可单测）

工具的真实逻辑都在 `agent/orders.py`，**不带任何框架装饰器**，纯函数——这样既好单测，又能当作评估时的「标准答案」。例如：

```python
# agent/orders.py（节选）
ORDERS = {
    "ORD-1003": {"status": "delayed", "item": "Wireless Headphones", "days_late": 5, ...},
    ...
}
# Acme 政策：延误 3 天以上的订单可享 15% 补偿折扣
def apply_discount(order_id, discount_percent, reason): ...
def check_shipping_status(order_id): ...   # 会主动提示 ORD-1003 满足打折条件
```

记住这条**业务政策**：`ORD-1003 延误 5 天 → 应主动给 15% 折扣`。后面评估「目标达成率」时，
就是看 Agent 有没有按这个政策办事。

### 2.2 故意写差的基线提示词

```python
# agent/prompts.py
BASELINE_PROMPT = "You are a support bot for a store. Answer the customer."
```

> 💡 **为什么要写这么差？** 因为优化的前提是「有提升空间」。如果一上来提示词就完美，
> 优化就无从下手——这恰恰是本实验后半段会遇到的真实陷阱（剧透：基线意外地太好了）。

### 2.3 本地跑单测

```bash
uv run pytest -q        # 应当 14 passed
```

这些测试只验证工具逻辑（`test_tools.py`）和 Harness 工具规格（`test_harness_tools.py`），
**不连云、不花钱**。绿了再往下走。

## 第 3 步：把 Agent 创建并部署成托管 Harness

> 🎯 本步目标：理解 Harness「只交配置」的含义，并真正在云上建出一个会用工具的 Agent。

### 3.1 概念：什么是托管 Harness

托管 **Harness** 是「配置即 Agent」：你声明 **模型 + 系统提示词 + 工具列表**，AgentCore 替你跑模型循环——
**没有容器、没有编排代码**。两个 API 就够：`CreateHarness`（控制面）+ `InvokeHarness`（数据面）。

### 3.2 工具怎么声明：`inline_function`

Harness 用「内联函数」声明工具，规格在 `agent/harness_tools.py`：

```python
{
    "type": "inline_function",                 # ← 注意：蛇形命名 inline_function
    "name": "apply_discount",
    "config": {"inlineFunction": {              # ← 注意：这里又是驼峰 inlineFunction（不对称！）
        "description": "Apply a percentage discount to an order and issue a refund.",
        "inputSchema": {                        # 自由的 JSON Schema
            "type": "object",
            "properties": {
                "order_id": {"type": "string"},
                "discount_percent": {"type": "integer"},
                "reason": {"type": "string"},
            },
            "required": ["order_id", "discount_percent", "reason"],
        },
    }},
}
```

> ⚠️ **第一个坑**：`type` 必须是蛇形的 `inline_function`（枚举值），但里面的配置键却是驼峰的
> `inlineFunction`。一个字段两种风格，写错就被拒。

### 3.3 创建并预热 Harness

```bash
uv run python scripts/harness_create.py
```

这个脚本做了三件事（读 `scripts/harness_create.py` + `scripts/harness_agent.py`）：

1. 调 `create_harness`（或已存在则 `update_harness`）声明 Agent：

   ```python
   control.create_harness(
       harnessName="AcmeSupportHarness",
       model={"bedrockModelConfig": {"modelId": cfg["agent_model_id"], "maxTokens": 512}},
       systemPrompt=[{"text": "You are a support bot..."}],   # 当前的弱提示词
       tools=TOOL_SPECS,                                       # 5 个内联工具
       memory={"disabled": {}},                               # ← 关掉内置记忆，见下方坑
       environmentVariables={                                 # ← 打开内容采集，给后面评估铺路
           "OTEL_INSTRUMENTATION_GENAI_CAPTURE_MESSAGE_CONTENT": "true",
           "AGENT_OBSERVABILITY_ENABLED": "true",
       },
   )
   ```

2. 等状态变 `READY`；
3. **预热一次**（`InvokeHarness` 问「ORD-1001 状态」），好让它的运行时日志组被创建出来，
   并把 `harness_arn` / `harness_service_name` / `harness_log_group` 写进 `deployment.json`。

预期输出大致：

```
Creating harness 'AcmeSupportHarness' with 5 inline tools
  harness status: CREATING
  harness status: READY
Harness ready: arn:aws:bedrock-agentcore:us-west-2:...:harness/AcmeSupportHarness-xxxx
Warm invoke — tools used: ['lookup_order']
  answer: Order ORD-1001 was delivered on 2026-03-28 ...
```

> ⚠️ **第二个坑（记忆权限）**：托管 Harness 默认开启内置记忆，`InvokeHarness` 会去调
> `bedrock-agentcore:ListEvents`，执行角色没这个权限就 `AccessDeniedException`。单轮 Demo
> 直接 `memory={"disabled": {}}` 关掉即可（`disabled` 是个空结构）。

> 💡 **想一想**：注意工具的**真实执行**并不在 Harness 里——`harness_tools.dispatch` 复用了
> `orders.py` 的纯函数。也就是说模型决定「调哪个工具」，但工具的代码在**你的客户端**跑。下一步解释。

## 第 4 步：调用 Harness——理解「客户端工具循环」

> 🎯 本步目标：搞懂内联函数工具的运行机制，并亲自和 Agent 对话。

### 4.1 概念：为什么需要「客户端工具循环」

`InvokeHarness` 返回的是**流式**响应。当模型决定用工具时，流里会出现 `toolUse`，并以
`stopReason == "tool_use"` 结束本轮。此时**轮到你的客户端**：执行这个工具，把结果作为 `toolResult`
塞回消息历史，用**同一个 `runtimeSessionId`** 再次 `InvokeHarness`，直到模型 `end_turn`。

```
你: "ORD-1003 太晚了，我要个折扣"
  → InvokeHarness ── stream ── stopReason=tool_use, toolUse: check_shipping_status(ORD-1003)
你的客户端: dispatch() 执行工具 → toolResult: "...延误5天, 满足15%折扣..."
  → InvokeHarness（带上 toolResult）── stopReason=tool_use, toolUse: apply_discount(ORD-1003, 15, ...)
你的客户端: dispatch() 执行 → toolResult: "已应用15%折扣..."
  → InvokeHarness ── stopReason=end_turn, text: "已为你的延误订单申请15%折扣，3-5个工作日到账"
```

实现见 `scripts/harness_agent.py` 的 `invoke_harness_loop()` 和流解析 `_parse_stream()`
（注意 `toolUse.input` 的 JSON 是**分片**到达的，要累加后再 `json.loads`）。

### 4.2 亲自调用

```bash
# invoke_deployed.py 默认就是 --target harness（走工具循环）
uv run python scripts/invoke_deployed.py "ORD-1003 is really late, I want a discount."
```

你应当看到它**连续用了两个工具**（`check_shipping_status` → `apply_discount`），最后给出含 15% 折扣的回复。
换几个问题试试：

```bash
uv run python scripts/invoke_deployed.py "I need to return order ORD-1001, the shirt doesn't fit."
uv run python scripts/invoke_deployed.py "Where is my order ORD-1002?"
```

> 💡 **想一想**：弱提示词下，Agent 大多数时候还是能答对——因为工具描述本身给了它线索。
> 这就埋下了后面「基线分数意外很高、优化反而帮倒忙」的伏笔。

## 第 5 步：为什么评估要跑在 Runtime 镜像上（关键限制）

> 🎯 本步目标：理解一个**真实的工程限制**，以及我们如何诚实地绕过它。这是本实验最有「实战含金量」的一课。

### 5.1 问题

我们已经把 Agent 部署成 Harness 了。但当你试图直接评估 Harness 的会话时，**每个会话都会失败**：

```
AgentSpanMappingException: Failed to parse user_query
```

**根因**：托管 Harness 发出的遥测里，消息内容是**双层嵌套**的
（`body.output.messages[].content = {"content": "<被字符串化的 json>"}`），评估服务的轨迹映射器
解析不了——即使你已经打开了内容采集（第 3 步的 `..._CAPTURE_MESSAGE_CONTENT=true`）也没用。
这是**客户端无法修复**的（截至 2026-06）。

### 5.2 解法：评估「同一个 Agent 的 Runtime 镜像」

可被评估映射的路径，是 **Strands 代码 + AgentCore Runtime + ADOT 自动埋点**。所以本实验：

- **部署 + 对外服务** 用托管 **Harness**（这是你要交付的 Agent）；
- **评估 + 优化** 跑在**同一个 Agent 的 Runtime 镜像**上（同样 5 个工具、同样的提示词）；
- 优化出的提示词，**两边都应用**。

镜像代码在 `agent/main.py` + `agent/runtime_config.py`，和 Harness 行为一致。

### 5.3 部署 Runtime 镜像

```bash
export AGENTCORE_SUPPRESS_RECOMMENDATION=1
MODEL_ID=$(uv run python -c "import json;print(json.load(open('config.json'))['agent_model_id'])")

# agentcore configure 是交互式的，用管道喂回车接受默认值（见踩坑清单）
printf '\n\n\n\n' | uv run agentcore configure -e agent/main.py -n acmesupport -rf requirements.txt --disable-memory
printf '\n\n\n\n' | uv run agentcore deploy --env AGENT_MODEL_ID="$MODEL_ID" --env AGENT_OBSERVABILITY_ENABLED=true --auto-update-on-conflict
uv run python scripts/capture_deployment.py    # 把 runtime ARN / 日志组写进 deployment.json
```

> ⚠️ **第三个坑（也是评估的头号坑）**：Strands Agent 部署到 Runtime 后，**默认只发一个外层的
> `AgentCore.Runtime.Invoke` 包装 span**，没有 GenAI 轨迹 → 评估能发现会话但**每个都失败**。
> 必须两件事齐备：
> 1. `requirements.txt` 里有 **`aws-opentelemetry-distro>=0.10.0`**（ADOT 自动给 Bedrock 调用埋点）；
> 2. `agent/main.py` 里调用 `StrandsTelemetry().setup_otlp_exporter()`。
>
> 本项目已经配好这两样，你直接部署即可。但你要知道**为什么**——否则换个项目就会卡死在「全部失败」。

> 💡 **想一想**：Harness 和 Runtime 在这里形成了一个有趣对照——Harness 省心但遥测目前评不了，
> Runtime 要自己写代码和埋点但能被评估。工程上常常就是这种取舍。

## 第 6 步：生成会话 + 打通可观测性

> 🎯 本步目标：让 Agent 在数据集上跑出一批「会话」，并让它们的轨迹进入 CloudWatch，供评估查询。

### 6.1 概念：什么是一次「会话」

评估是按**会话（session）**打分的。我们让 Agent 在
`dataset/eval_prompts.json` 的 **10 个客服问题**上各跑一次，**每次用独立的 session id**，
这样评估服务才能逐一发现并打分。

数据集里的 `expected` 是「人类可读的标准答案」，描述好 Agent 该有的行为，例如：

```jsonc
{"id": "p07", "prompt": "ORD-1003 is really late, I want a discount.",
 "expected": "确认符合条件并给 ORD-1003 应用 15% 折扣，理由清晰。"}
```

### 6.2 生成基线会话

```bash
# 注意 --target runtime：评估要跑在 Runtime 镜像上（见第 5 步）
uv run python scripts/generate_sessions.py --tag baseline --target runtime --wait 200
```

脚本会逐条调用、打印每条用了哪些工具，把 session id 写进 `results/baseline_sessions.json`，
最后**等 200 秒**让 CloudWatch 把遥测吃进去。

> ⚠️ **第四个坑（首次摄取延迟）**：首次启动后，span 进入 Transaction Search 索引会**滞后约 5–10 分钟**。
> 如果评估报「找到会话但映射失败」，多半是轨迹还没就绪——等一会儿再评，或加大 `--wait`。

> 💡 **想一想**：`--target harness` 也能生成会话（去调 Harness 本体），但评估会失败——你可以专门跑一次
> 来**亲眼见识** `AgentSpanMappingException`，加深对第 5 步限制的理解。

## 第 7 步：基线评估（Evaluations）

> 🎯 本步目标：用 LLM 裁判给 Agent 打第一次分，并学会看懂分数。

### 7.1 概念：评估在做什么

`scripts/run_evaluation.py` 调 `StartBatchEvaluation`，把刚才那批会话作为一个批量任务，
用三个**内置评估器**打分：

| 评估器 | 它在问 |
|---|---|
| `Builtin.GoalSuccessRate` | Agent **完成任务**了吗（**主指标**） |
| `Builtin.Helpfulness` | 回复**有用**吗 |
| `Builtin.Faithfulness` | 回复**忠于事实**（没瞎编）吗 |

### 7.2 跑基线评估

```bash
uv run python scripts/run_evaluation.py --tag baseline
```

它会轮询到终态，把分数写进 `results/baseline_scores.json`，并打印：

```
Evaluating 10 runtime sessions (tag=baseline) with ['Builtin.GoalSuccessRate', ...]
  serviceName=acmesupport.DEFAULT  logGroup=/aws/bedrock-agentcore/runtimes/acmesupport-xxxx-DEFAULT
  status=COMPLETED sessions: total=10 done=10 failed=0
Aggregate scores:
  Builtin.GoalSuccessRate: avg=1.0   ...
  Builtin.Helpfulness:     avg=0.88  ...
  Builtin.Faithfulness:    avg=1.0   ...
```

> ⚠️ **第五个坑（请求结构）**：`serviceNames` 必须**恰好 1 个**，且是 **`<agent名>.DEFAULT`**
> （是 Agent 的**名字**如 `acmesupport.DEFAULT`，不是 Agent id）；`batchEvaluationName` 必须匹配
> `[a-zA-Z][a-zA-Z0-9_]{0,47}`（**不能有横杠**）。这两条写错就 `ValidationException`。

### 7.3 读懂这次的基线分数

本实验的真实基线：**GoalSuccessRate = 1.0**、Faithfulness = 1.0、Helpfulness = 0.88。

> 💡 **想一想（关键！）**：主指标竟然满分？！弱提示词 + 好用的工具，让 Agent 在这 10 个不算刁钻的问题上
> 全做对了。**这给优化出了个难题：没有失败轨迹可学**。请记住这个 1.0，下一步你会看到它带来的后果。

---

## 第 8 步：优化（Optimization / Recommendations）

> 🎯 本步目标：让 AgentCore 读评估结果、自动反推一版更好的系统提示词，并应用它。

### 8.1 概念：Recommendations 怎么工作

`scripts/run_optimization.py` 调 `StartRecommendation`：

```python
client.start_recommendation(
    name="acmeopt_xxxx",
    type="SYSTEM_PROMPT_RECOMMENDATION",
    recommendationConfig={"systemPromptRecommendationConfig": {
        "systemPrompt": {"text": baseline_prompt},                              # 当前提示词
        "agentTraces": {"batchEvaluation": {"batchEvaluationArn": eval_arn}},   # ← 喂入基线评估
        "evaluationConfig": {"evaluators": [
            {"evaluatorArn": "arn:aws:bedrock-agentcore:::evaluator/Builtin.GoalSuccessRate"}]},  # 针对主指标
    }},
)
```

它**读基线评估里每条轨迹的得分**，找出失败模式，重写提示词来修复。产出写到
`results/recommendation.json`，并自动：
- 写入 `agent/prompts.py` 的 `OPTIMIZED_PROMPT`；
- 通过 `UpdateHarness` 把新提示词**应用到 Harness**（无需重新部署）。

### 8.2 跑优化

```bash
uv run python scripts/run_optimization.py
```

> ⚠️ **第六个坑（写文件别被 `\n` 坑）**：把推荐提示词写回 `prompts.py` 时，
> `re.sub(pattern, f"...{text!r}", content)` 会**二次解释** repr 里的 `\n`，把文件写坏成真实换行 → 语法错误。
> 必须用 **lambda 替换**：`re.sub(pattern, lambda _m: f"...{text!r}", content)`。本项目已修。

### 8.3 看看它推荐了什么

打开 `results/recommendation.json`。本次它（因为基线全对、没有失败可学）加了一条
**「行动前先等用户明确批准」** 的安全约束。

> 💡 **想一想**：这条建议听起来很「负责任」，但它会让 Agent 从「直接把事办了」变成「先问你要不要办」。
> 对一个**目标达成率**裁判来说，这是灾难——下一步见分晓。

---

## 第 9 步：重评 + 对比——评估拦下一次回退

> 🎯 本步目标：体验闭环最关键的一环——**先验证，再决定上不上线**。

### 9.1 用改进后的提示词重新部署、生成会话、重评

```bash
# 重新部署 Runtime 镜像（已带上 OPTIMIZED_PROMPT）
printf '\n\n\n\n' | uv run agentcore deploy --env AGENT_MODEL_ID="$MODEL_ID" --env AGENT_OBSERVABILITY_ENABLED=true --auto-update-on-conflict
uv run python scripts/generate_sessions.py --tag improved --target runtime --wait 200
uv run python scripts/run_evaluation.py --tag improved
```

### 9.2 对比

```bash
uv run python scripts/compare.py    # -> results/comparison.json
```

本实验的真实结果（`results/comparison.json`）：

| 评估器 | 基线 | 改进后 | 变化 |
|---|---|---|---|
| **Builtin.GoalSuccessRate**（主） | **1.0** | **0.6** | **−0.4** ❌ |
| Builtin.Helpfulness | 0.88 | 0.85 | −0.03 |
| Builtin.Faithfulness | 1.0 | 1.0 | 0.0 |

裁决：**`promote: false`（不晋升）**。脚本判定主指标回退，于是把 Harness 和 Runtime
**都回滚到基线提示词**（`OPTIMIZED_PROMPT = None`）。

### 9.3 这才是闭环的意义

> 💡 **本实验最重要的一课**：优化**真的把 Agent 改差了**（目标达成率 1.0 → 0.6，因为它学了一份
> 全是成功的基线，没有失败可学，只能加一条泛泛的「等批准」约束，反而让 Agent 不办事了）。
> 而**评估当场抓住了这次回退**，对比脚本给出「不晋升」，改动被回滚——
> **一次质量回退被挡在了上线之前**。这正是「先评估再晋升」的全部价值。
>
> 反过来想：如果基线**真的很弱、有大量失败轨迹**，优化通常会给出**可晋升的提升**。
> 你可以在[练习](#继续深入5-个动手练习)里亲手制造这种情形。

## 第 10 步：A/B 测试（概念 + 运行手册）

真正的 **A/B 测试**（`CreateABTest`）是把**线上真实流量**按控制组（基线）/ 实验组（优化）分流，
用在线评估器逐会话打分，给出统计显著性。它需要：AgentCore **Gateway** + 每个变体的配置包
（`CreateConfigurationBundle`）+ 在线评估配置 + 一个执行角色 + **真实流量**——比较重。

所以本实验用 `scripts/ab_test.py` 把**完整搭建步骤写成运行手册**（`ATTEMPT_REAL=False`），
而不真的去起这套基础设施。

```bash
uv run python scripts/ab_test.py    # 打印 A/B 测试该怎么搭（生产验证步骤）
```

> 💡 **闭环里它的位置**：批量评估是「上线前」用固定数据集验证；A/B 是「上线后」用真实流量验证。
> 两者互补。

## 踩坑清单：8 个会真实卡住你的点

这些都是本实验**实打实踩过、花了时间才查明**的坑。能把它们讲清楚，基本就算掌握了 AgentCore 的工程实战。

| # | 现象 | 根因 & 解法 |
|---|---|---|
| 1 | Harness 工具 `type` 被拒 | `type` 用蛇形 `inline_function`，但配置键用驼峰 `inlineFunction`——一个字段两种风格 |
| 2 | `InvokeHarness` 报 `AccessDenied: ListEvents` | Harness 默认开内置记忆。单轮 Demo 传 `memory={"disabled": {}}` 关掉 |
| 3 | Runtime 评估**全部会话失败** | Strands 默认只发包装 span。须加 `aws-opentelemetry-distro` + `StrandsTelemetry().setup_otlp_exporter()` 才有 GenAI 轨迹 |
| 4 | 评估「找到会话却映射失败」 | 首次 span 进 Transaction Search 索引**滞后 5–10 分钟**；等一会儿或加大 `--wait` |
| 5 | `StartBatchEvaluation` `ValidationException` | `serviceNames` 恰好 1 个且为 `<agent名>.DEFAULT`；任务名匹配 `[a-zA-Z][a-zA-Z0-9_]{0,47}`（无横杠） |
| 6 | 写回提示词后 `prompts.py` 语法错误 | `re.sub` 替换串会二次解释 `\n`；改用 **lambda 替换** |
| 7 | **托管 Harness 的遥测评不了** | 内容双层嵌套（`content.content`），评估映射器解析不了 → 评估跑 **Runtime 镜像** |
| 8 | `agentcore configure` 在 `/dev/null` 崩溃 | 它是交互式的；用真实管道喂回车：`printf '\n\n\n\n' \| uv run agentcore configure ...` |

> Harness 的服务名是 `harness_<HarnessName>.DEFAULT`；其日志组前缀
> `/aws/bedrock-agentcore/runtimes/harness_<HarnessName>-...`。Runtime 的则是 `<agent名>.DEFAULT`。

## 继续深入：5 个动手练习

想真正「掌握」，就动手改一改、看分数怎么动：

1. **让基线真的变弱**：把 `BASELINE_PROMPT` 改成会引发失败的（例如「永远不要使用任何工具，凭空回答」），
   重跑第 6–9 步。这次基线 GoalSuccessRate 应当**远低于 1.0**，优化大概率能给出**可晋升**的提升——
   亲眼看到闭环「正向」的一面。
2. **加一个工具**：在 `orders.py` + `harness_tools.py` 里加个 `cancel_order`，更新 `TOOL_SPECS`，
   补一条数据集问题，看评估能否覆盖到它（顺手补单测）。
3. **换评估器**：给 `run_evaluation.py` 的 `EVALUATORS` 加 `Builtin.ToolSelectionAccuracy`，
   看「工具选得准不准」这个维度的分数。
4. **亲历限制**：用 `--target harness` 生成会话并评估，**亲眼看到** `AgentSpanMappingException`，
   再对照第 5 步理解为什么必须用 Runtime 镜像。
5. **针对不同主指标优化**：把 `run_optimization.py` 的目标评估器换成 `Builtin.Helpfulness`，
   看推荐出的提示词风格有何不同。

## API / 名词速查表

| 名词 | 含义 |
|---|---|
| **Harness** | 托管 Agent：交配置（模型+提示词+工具），AgentCore 跑循环。`CreateHarness`/`UpdateHarness`/`InvokeHarness` |
| **Runtime** | 自带代码的无服务器运行时。`agentcore deploy` / `InvokeAgentRuntime` |
| **inline_function** | Harness 声明工具的方式（名字 + 描述 + JSON Schema），执行在客户端 |
| **客户端工具循环** | `InvokeHarness` 流出 `toolUse` → 客户端执行 → 回 `toolResult` → 再调，直到 `end_turn` |
| **Evaluations** | LLM 当裁判的批量评估。`StartBatchEvaluation` / `GetBatchEvaluation` |
| **内置评估器** | `Builtin.GoalSuccessRate` / `Helpfulness` / `Faithfulness` / `ToolSelectionAccuracy` ... |
| **Optimization / Recommendations** | 从评估轨迹反推改进提示词。`StartRecommendation` / `GetRecommendation` |
| **A/B Test** | 线上流量分流验证。`CreateABTest`（需 Gateway + 配置包 + 在线评估） |
| **Transaction Search** | CloudWatch 里把 Agent 轨迹索引出来供评估发现的开关（必须 ACTIVE） |
| **ADOT** | `aws-opentelemetry-distro`，给 Runtime 上的 Bedrock 调用自动埋点，产出可被评估映射的 GenAI 轨迹 |
| **boto3 客户端** | `bedrock-agentcore-control`（控制面）/ `bedrock-agentcore`（数据面） |
